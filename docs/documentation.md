## 1. Backend, Frontend и База данных

Любое веб-приложение, включая МКБ-МИС, состоит из трёх основных частей. Удобно представить
их как ресторан:

- **Frontend** — зал и официант: то, что видит гость.
- **Backend** — кухня: где готовят, но гость её не видит.
- **База данных** — склад продуктов: где всё хранится.

### 1.1. Frontend (клиентская часть)

Это то, с чем работает пользователь — врач. Открывается в **браузере**.

- Технологии: **HTML** (структура страницы), **CSS** (оформление), **JavaScript** (поведение:
  кнопки, поиск, формы).
- Задача: показать данные красиво и удобно, принять действия пользователя (ввод логина,
  поиск пациента, заполнение формы) и отправить их на сервер.
- Frontend **не хранит** данные постоянно и **не содержит** бизнес-правил — он только
  отображает и отправляет/получает.

В МКБ-МИС это папка `client/`.

### 1.2. Backend (серверная часть)

Это «мозг» системы, работающий на сервере. Пользователь его не видит.

- Технологии: **Python** + REST API.
- Задачи: принять запрос от клиента, проверить права доступа (роль врача/админа), выполнить
  бизнес-логику (например, проверить, что размер камня больше нуля), обратиться к базе данных
  и вернуть ответ.
- Именно backend решает, **что можно, а что нельзя**, и отвечает за безопасность.

В МКБ-МИС это папка `server/`.

### 1.3. База данных (БД)

Место, где данные хранятся **постоянно** — даже после закрытия программы.

- Технология: **PostgreSQL**.
- В МКБ-МИС четыре таблицы (см. `schema.sql`):
  - `users` — врачи и администраторы;
  - `patients` — пациенты;
  - `examinations` — обследования;
  - `stones` — камни.
- Таблицы связаны: у пациента есть лечащий врач (`patients.id_user → users`), у камня и
  обследования — пациент (`id_patient → patients`). Это связи «один ко многим»: один врач —
  много пациентов, один пациент — много камней и обследований.

### 1.4. Как они связаны

Frontend сам в базу не ходит — только через backend. Поток одного действия:

```
Врач в браузере          Сервер (Python)            База данных
   (Frontend)               (Backend)               (PostgreSQL)
       │                        │                         │
       │  1. HTTP-запрос        │                         │
       │  «найди пациента» ───▶ │                         │
       │                        │  2. SQL-запрос ───────▶ │
       │                        │                         │
       │                        │  3. строки из таблиц ◀─ │
       │  4. ответ в JSON  ◀─── │                         │
       │                        │                         │
   5. показывает список         │                         │
```

Такое разделение удобно: над каждой частью может работать свой человек, а заменить одну часть
(например, переписать интерфейс) можно, не трогая остальные.

## 2. Основные API простыми словами

### 2.1. Что такое API

**API** (Application Programming Interface) — это набор «команд», по которым frontend просит
backend что-то сделать. Это как меню в ресторане: гость не идёт на кухню, а выбирает из
понятного списка блюд. Точно так же клиент не лезет в базу, а вызывает готовые команды сервера.

В проекте используется **REST API** — обмен идёт по протоколу **HTTP** (тому же, что и сайты).

Каждый запрос состоит из:

- **Метода** — что хотим сделать:
  - `GET` — получить данные (прочитать);
  - `POST` — создать новое;
  - `PUT` — изменить существующее;
  - `DELETE` — удалить.
- **Адреса (эндпоинта)** — над чем действуем, например `/patients`.
- **Тела** (для POST/PUT) — данные в формате **JSON**.
- В ответ приходит **код состояния** (например, `200` — успех, `401` — нет доступа,
  `404` — не найдено) и данные в JSON.

### 2.2. Главные эндпоинты МКБ-МИС

| Действие | Команда | Простыми словами |
|---|---|---|
| Вход в систему | `POST /auth/login` | «Проверь логин и пароль, впусти меня» |
| Список пациентов | `GET /patients` | «Покажи всех пациентов» |
| Поиск пациента | `GET /patients?search=Иванов` | «Найди пациентов по ФИО или номеру карты» |
| Карточка пациента | `GET /patients/1` | «Покажи данные пациента №1» |
| Добавить пациента | `POST /patients` | «Заведи нового пациента» |
| Изменить пациента | `PUT /patients/1` | «Сохрани изменения в карточке» |
| Камни пациента | `GET /patients/1/stones` | «Покажи камни пациента №1» |
| Добавить камень | `POST /patients/1/stones` | «Запиши новый камень» |
| Добавить обследование | `POST /patients/1/examinations` | «Запиши новое обследование» |
| Создать аккаунт врача | `POST /users` | «Заведи нового врача» (только администратор) |

Полное описание полей запросов и ответов — в `client/README.md`.

### 2.3. Пример

Запрос на вход:

```
POST /auth/login
{ "username": "doctor_ivanov", "password": "hash123" }
```

Ответ сервера:

```
200 OK
{
  "token": "...",
  "user": { "id_user": 1, "full_name": "Иванов Иван Иванович", "role": "doctor" }
}
```

`token` — это «пропуск». Клиент прикладывает его ко всем следующим запросам, чтобы сервер
понимал, кто обращается.

## 3. Архитектура больших проектов

### 3.1. Зачем нужна архитектура

Когда кода много, без структуры он превращается в «кашу», где всё связано со всем и любое
изменение ломает другое. Архитектурные шаблоны задают **правила разделения**: где хранить
данные, где логику, где интерфейс. Это упрощает разработку в команде, тестирование и развитие.

Главная идея большинства шаблонов — **разделение ответственности**: каждая часть отвечает за
своё.

### 3.2. MVC (Model — View — Controller)

Классический шаблон, делит приложение на три слоя:

- **Model (Модель)** — данные и правила работы с ними (таблицы, бизнес-логика).
- **View (Представление)** — то, что видит пользователь (страницы, формы).
- **Controller (Контроллер)** — посредник: принимает действия пользователя, обращается к
  модели, выбирает, что показать.

Поток: пользователь нажимает кнопку во **View** → **Controller** обрабатывает → берёт/меняет
данные в **Model** → **View** обновляется.

Аналогия: View — витрина, Controller — продавец, Model — склад.

### 3.3. MVVM (Model — View — ViewModel)

Развитие MVC, популярно в современных интерфейсах (Angular, Vue, WPF). Отличие — слой
**ViewModel** и **связывание данных (data binding)**:

- **Model** — данные.
- **View** — интерфейс.
- **ViewModel** — «прослойка» между ними: хранит состояние экрана и автоматически
  синхронизирует его с View.

Главное отличие от MVC: при MVVM не нужно вручную обновлять интерфейс — изменил данные в
ViewModel, и View меняется сам (и наоборот). Это удобно для сложных динамических интерфейсов.

### 3.4. MVC и MVVM кратко

| | MVC | MVVM |
|---|---|---|
| Третий слой | Controller (управляет потоком) | ViewModel (хранит состояние экрана) |
| Связь View и данных | Ручная | Автоматическая (data binding) |
| Где чаще | Серверные веб-фреймворки | Современные клиентские фреймворки |

### 3.5. Другие подходы в больших проектах

- **Клиент-серверная архитектура** — основа МКБ-МИС: клиент и сервер — отдельные программы,
  общаются по сети.
- **Многослойная (layered)** — деление на слои: представление → логика → доступ к данным.
- **Микросервисы** — приложение разбито на много маленьких независимых сервисов (для очень
  крупных систем).

### 3.6. Как это устроено в МКБ-МИС

Проект построен по **клиент-серверной** архитектуре, а роли слоёв распределены так:

| Слой | В проекте |
|---|---|
| View (представление) | `client/` — HTML/CSS и отрисовка в `app.js` |
| Логика / Controller | `server/` — Python, обработка REST-запросов |
| Model / данные | PostgreSQL по схеме `schema.sql` |

Связь между клиентом и сервером изолирована: на клиенте весь обмен собран в одном файле
`client/js/api.js`. Поэтому интерфейс и сервер можно разрабатывать и менять независимо — это и
есть практическая польза от продуманной архитектуры.
