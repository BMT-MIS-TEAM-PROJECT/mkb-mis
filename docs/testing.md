# Тестирование проекта МКБ-МИС

**Исполнитель:** Базылбеков Бектур — тестирование
**Инструменты:** браузер (+ DevTools), Postman
**Формат:** Markdown

Документ состоит из двух частей: (1) методы тестирования систем, состоящих из большого
числа узлов, и (2) набор тестов, опирающихся на архитектуру системы, а не на её код.

---

## Часть 1. Методы тестирования многоузловых систем

### 1.1. Что считается «узлом»

МКБ-МИС — это не одна программа, а несколько взаимодействующих по сети компонентов (узлов):

- **Клиент** — веб-интерфейс врача (HTML/CSS/JS в браузере);
- **Сервер** — REST API (обработка запросов, бизнес-логика, авторизация);
- **База данных** — PostgreSQL (таблицы `users`, `patients`, `examinations`, `stones`);
- **Канал связи** — HTTP/REST между клиентом и сервером.

Чем больше узлов, тем меньше ошибок находится «внутри» одного модуля и тем больше — на
**стыках** между узлами: несовпадение форматов данных, неверные коды ответов, потеря связи,
рассинхронизация версий API. Поэтому акцент тестирования смещается с логики отдельного модуля
на проверку взаимодействий.

### 1.2. Уровни тестирования (пирамида тестов)

Тесты выстраиваются по уровням — от мелких и быстрых к крупным и медленным:

| Уровень | Что проверяет | Пример в МКБ-МИС |
|---|---|---|
| Модульное (unit) | Отдельную функцию/модуль изолированно | Функция хеширования пароля, валидация `size_mm > 0` |
| Интеграционное | Взаимодействие двух-трёх узлов | Сервер ↔ БД: запрос «список пациентов» возвращает данные из БД |
| Системное / E2E | Всю цепочку узлов целиком | Браузер → API → БД → ответ → отрисовка списка |
| Приёмочное | Соответствие требованиям заказчика | «Врач может найти пациента по ФИО» |

**Принцип пирамиды:** модульных тестов должно быть много (они дешёвые и быстрые),
интеграционных — меньше, сквозных (E2E) — мало (они дорогие и хрупкие). Это удерживает
прогон тестов быстрым и стабильным.

### 1.3. Методы по уровню доступа к коду

- **Чёрный ящик** — тестировщик не знает внутреннего устройства, проверяет систему через её
  интерфейсы (REST API, UI). Главный метод для проверки взаимодействия узлов.
- **Белый ящик** — тесты пишутся с учётом исходного кода (ветвления, граничные значения внутри функций).
- **Серый ящик** — частичное знание устройства: например, тестировщик знает схему БД и контракт API,
  но не реализацию сервера. Это основной режим для данного проекта.

### 1.4. Методы, критичные именно для многоузловых систем

1. **Интеграционное тестирование стыков.** Проверка, что узлы корректно обмениваются данными:
   правильные эндпоинты, форматы JSON, коды состояния HTTP.

2. **Контрактное тестирование (contract testing).** Узлы разрабатываются разными людьми
   (клиент и сервер — разные исполнители). Фиксируется «контракт» — описание эндпоинтов,
   полей запроса и ответа. Тесты проверяют, что сервер отдаёт ровно то, что ожидает клиент.
   Удобно автоматизировать в **Postman** (коллекция с проверками полей и кодов).

3. **Изоляция узлов через заглушки.** Когда один узел ещё не готов, его заменяют:
   - **stub / mock** — заглушка вместо недостающего сервера (в клиенте МКБ-МИС это режим `USE_MOCK = true`);
   - **driver** — программа, имитирующая вызывающий узел.
   Это позволяет тестировать узлы независимо («снизу вверх», «сверху вниз» или комбинированно).

4. **Сквозное (E2E) тестирование через браузер.** Проверка пользовательских сценариев целиком:
   вход → поиск → открытие карточки → добавление обследования.

5. **Дымовое (smoke) и регрессионное тестирование.** Дымовое — быстрая проверка, что система
   «жива» после сборки (открывается вход, отвечает API). Регрессионное — что новые изменения
   не сломали уже работавший функционал.

6. **Нагрузочное и стресс-тестирование.** При множестве узлов важны узкие места: время ответа
   API при многих одновременных запросах, поведение под пиковой нагрузкой. Инструменты:
   Postman Runner, JMeter, k6.

7. **Тестирование отказоустойчивости.** Что происходит при отказе узла: сервер недоступен —
   клиент показывает понятную ошибку, а не «зависает»; БД недоступна — сервер возвращает 5xx,
   а не падает. Крайний вариант — chaos-тестирование (намеренный вывод узлов из строя).

8. **Тестирование безопасности.** Проверка авторизации между узлами: запрос без токена
   отклоняется (401), врач не имеет доступа к админ-функциям (403), пароли не передаются в открытом виде.

9. **Наблюдаемость как часть тестирования.** В распределённой системе ошибку трудно
   локализовать без логов и мониторинга. Логирование запросов на сервере — обязательная часть
   тестируемости.

### 1.5. Инструменты

- **Postman** — ручная и автоматическая проверка REST API: коллекции запросов, переменные
  окружения (`base_url`, `token`), скрипты-проверки на JavaScript (вкладка *Tests*), запуск
  коллекции целиком (*Collection Runner*).
- **Браузер + DevTools** — проверка UI, вкладка *Network* (какие запросы уходят и что приходит),
  *Console* (ошибки JS), эмуляция медленной сети и оффлайна.

### 1.6. Стратегия для МКБ-МИС

Комбинированный подход: модульные тесты на сервере (валидация, бизнес-логика) → контрактные и
интеграционные тесты API в Postman → сквозные сценарии в браузере → отдельно проверки
безопасности и отказоустойчивости. Так каждый узел тестируется и по отдельности, и в связке.

## Часть 2. Тесты на основе архитектуры (без привязки к коду)

Тесты ниже — **чёрный ящик**: они опираются на архитектуру и контракт системы (REST API из
`client/README.md` и схему БД `schema.sql`), а не на реализацию. Их можно выполнять, пока код
ещё не написан, — они задают ожидаемое поведение.

Базовые предусловия для API-тестов: переменная окружения `base_url` (например,
`http://localhost:8000/api`), для защищённых запросов — токен из теста авторизации.

Формат тест-кейса: **ID · цель · действие · ожидаемый результат**.

### 2.3. Контрактные проверки (стык клиент ↔ сервер)

Цель — убедиться, что ответ сервера соответствует ожиданиям клиента. В Postman, вкладка *Tests*:

```javascript
// Проверка контракта для GET /patients
pm.test("Код ответа 200", () => pm.response.to.have.status(200));
pm.test("Ответ — массив", () => Array.isArray(pm.response.json()));
pm.test("Есть обязательные поля", () => {
  const p = pm.response.json()[0];
  pm.expect(p).to.have.all.keys(
    "id_patient", "id_user", "full_name", "anamnesis",
    "doctor_name", "stones_count", "examinations_count"
  );
});
```

```javascript
// Сохранение токена после входа для последующих запросов
pm.test("Токен получен", () => {
  const body = pm.response.json();
  pm.expect(body).to.have.property("token");
  pm.environment.set("token", body.token);
});
```

## Итог

Для многоузловой системы основной риск — на стыках узлов, поэтому ядро стратегии составляют
контрактное и интеграционное тестирование API (Postman) плюс сквозные сценарии в браузере,
дополненные проверками ролей, безопасности и отказоустойчивости. Приведённые тест-кейсы
сформулированы как «чёрный ящик» по архитектуре и контракту, поэтому применимы независимо от
конкретной реализации сервера.
